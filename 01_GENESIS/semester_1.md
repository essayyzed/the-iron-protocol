#### ‚å®Ô∏è **Touch Typing Practice (15 minutes/day)**

**Tools:**

#### üêß **Linux as Primary OS (30-60 minutes/day)**

**Action Steps:**
1. **Install Linux** (dual boot or primary):
   - Ubuntu 22.04+ (beginner-friendly)
   - Fedora (intermediate)
   - Pop!_OS (good for students)

2. **Do ALL your coursework in Linux**
   - Write assignments in LibreOffice/LaTeX
   - Code in vim/nano/VSCode
   - Browse, email, everything

3. **Learn Terminal Basics:**

```bash
# Navigation
ls          # list files
cd          # change directory
pwd         # print working directory

# File operations
cp          # copy
mv          # move/rename
rm          # remove (CAREFUL!)
mkdir       # make directory
touch       # create empty file

# Viewing files
cat         # display file content
less        # view file with scrolling
head        # first lines
tail        # last lines

# Search
grep        # search in files
find        # find files
man         # manual pages
```

**Weekly Challenge:** Learn 3 new terminal commands each week.

#### ‚ùì **Write Good Questions (1 per week minimum)**

**The Protocol:**
1. **Write** ‚Üí Articulate your problem clearly
2. **Narrow** ‚Üí What specifically are you stuck on?
3. **Search** ‚Üí Google, Stack Overflow, documentation
4. **Ask** ‚Üí If still stuck, ask with context

**Template:**
```
I'm trying to: [goal]
I expected: [expected result]
I got: [actual result]
I've tried: [solutions attempted]
My code: [relevant snippet]
Error message: [if any]
```

**Non-negotiable:** You must include a **minimal reproducible example**:
- smallest input that breaks your code
- smallest snippet that reproduces the bug
If your question cannot be reproduced, it is not a ‚Äúgood question‚Äù yet.

#### üß† The 20-Minute Struggle (Structured)
Before asking for help, do this:

- **5 min:** write expected vs actual output
- **5 min:** read the error *slowly* and search the exact message
- **5 min:** reduce to the smallest failing case
- **5 min:** run 1‚Äì2 experiments (print statements / logging / simpler inputs)

When you ask, include what you tried + what changed.

## üì¶ Mini Projects (End of Semester Targets)

### ‚úÖ Definition of Done (Applies to All Semester 1 Projects)
For every project:
- GitHub repo folder + clean README
- clear run instructions
- at least **5 test cases** (manual is okay)
- include sample input/output in README
- include a `bugs.md` listing:
  - 2 bugs you hit
  - how you found them
  - how you fixed them

### Project 1: CLI Calculator
**Difficulty:** Beginner  
**Time:** 1-2 weeks

**Requirements:**
- Command-line interface
- Basic operations: +, -, *, /
- Handle invalid input gracefully
- Written in your course language (C++/Python/Java)

**Learning outcomes:** Basic I/O, conditionals, loops, error handling

---

### Project 2: Text-Based Game
**Difficulty:** Beginner  
**Time:** 2-3 weeks

**Options:**

**Requirements:**
- User input
- Game logic
- Win/lose conditions
- Replay functionality

**Learning outcomes:** Control flow, functions, data structures (arrays/lists)

## ‚ö†Ô∏è Common Mistakes

### ‚ùå **Skipping typing practice**
*"I'll do it later"* = never. Start now.

### ‚ùå **Staying in Windows/Mac comfort zone**
You won't learn problem-solving without friction.

### ‚ùå **Copying code without understanding**
Type it out. Break it. Fix it. That's how you learn.

### ‚ùå **Not asking questions**
Fear of "looking dumb" is how you stay dumb. Ask.

### ‚ùå **Waiting for perfection**
Your first projects will be ugly. That's okay. Ship them anyway.
